SPEC_COMPLETED: 177-drop-service-annotations-and-centralize-handler-discovery.md
WHAT_WAS_DONE: Deleted the three class-level `@ToolService` / `@PromptService` / `@ResourceService` marker annotations from `mocapi-api` and centralized handler discovery through a new `HandlerMethodsCache` record in `mocapi-server/src/main/java/com/callibrity/mocapi/server/autoconfigure/HandlerMethodsCache.java`. The cache's static `scan(ConfigurableListableBeanFactory, List<Class<? extends Annotation>>)` walks every bean definition once (via `getBeanNamesForType(Object.class, false, false)`), uses `beanFactory.getType(name, false)` to read each candidate's declared type *without forcing instantiation*, filters to beans whose class hosts at least one of the four method-level annotations, then for each surviving bean does a single `getBean(name)` + `AopUtils.getTargetClass` unwrap and groups every `(beanName, bean, method)` triple by annotation type into an immutable `Map<Class<? extends Annotation>, List<BeanMethod>>`. The pre-filter avoids the circular-init risk that a naive `getBeansOfType(Object.class)` would have (the cache itself is a Spring bean, and forcing every other bean's construction from inside its factory method would cascade back into `McpToolsService`/`McpPromptsService`/`McpResourcesService` — those depend on the cache). The cache bean is registered in `MocapiServerAutoConfiguration` with `@ConditionalOnMissingBean(HandlerMethodsCache.class)` and four annotation types passed in: `McpTool`, `McpPrompt`, `McpResource`, `McpResourceTemplate`. Each of the four per-kind handler factories (`CallToolHandlers`, `GetPromptHandlers`, `ReadResourceHandlers`, `ReadResourceTemplateHandlers`) lost its `discover(bean, …)` method and now exposes only a public `build(Object bean, Method method, … deps)` that constructs exactly one handler from a single `(bean, method)` pair. Each per-kind handler autoconfig (`MocapiServerToolsAutoConfiguration`, `MocapiServerPromptsAutoConfiguration`, `MocapiServerResourcesAutoConfiguration`) now injects `HandlerMethodsCache`, consumes its slice via `cache.forAnnotation(McpX.class).stream().map(bm -> X.build(bm.bean(), bm.method(), …)).toList()`, and (for prompts/resources) walks the resulting handler list to register completion candidates with `McpCompletionsService` exactly as before. The three handler autoconfigs flipped their ordering hint from `@AutoConfiguration(before = MocapiServerAutoConfiguration.class)` to `@AutoConfiguration(after = MocapiServerAutoConfiguration.class)` so the cache bean is registered before them. Per-handler startup log lines were reshaped to the single-line form `Registered MCP tool: "greet" (bean "greetingTool")` now that there's no per-bean "registering…" outer group. The AOT processor (`MocapiServicesAotProcessor`) was rewritten to discover candidate beans the same way — a bean contributes AOT hints iff its declared class has at least one `@McpTool` / `@McpPrompt` / `@McpResource` / `@McpResourceTemplate` method. The 21 application / example / test / conformance files that carried `@ToolService` / `@PromptService` / `@ResourceService` were stripped (annotation + `import` both) via a two-stage `sed` pass (top-level `@ToolService$` lines first, then indented `^ *@ToolService$` for nested static test fixtures). The six handler-kind unit tests that previously called the removed `discover(…)` helpers (`CallToolHandlerTest`, `McpToolsServiceTest`, `GetPromptHandlerTest`, `ReadResourceHandlerTest`, `ReadResourceTemplateHandlerTest`, `MocapiServicesAotProcessorTest`) were updated to iterate `MethodUtils.getMethodsListWithAnnotation(target.getClass(), McpX.class)` themselves and map each match through the new `build(…)` — each test file grew a small private `createHandlers(Object target)` helper. The AOT processor test dropped its `@ToolService` / `@PromptService` / `@ResourceService` fixture annotations since the processor now gates on method-level annotations alone. Docs were rewritten: `README.md` (three quickstart snippets), `docs/tools-guide.md`, `docs/prompts-guide.md`, `docs/resources-guide.md`, `docs/handlers.md` (four "Discovery lives in … #discover" paragraphs swapped to "Handlers are built by mapping each (bean, method) pair from the central HandlerMethodsCache through X.build(…) in …AutoConfiguration"), `docs/externalizing-metadata.md`, `docs/validation.md` (three fixture snippets), `docs/native-image-hints.md` (AOT processor description and the `mocapi.api.*` count dropped from 5 to 3), and `docs/architecture.md` (startup-logging sample collapsed to single-line form). `CHANGELOG.md` gained a new top-of-`### Breaking changes` entry under `## [Unreleased]` explaining the drop with a migration snippet (`-@ToolService` → remove, keep `@Component`) and noting that users who already register via `@Bean` need no change.

NOTES: HandlerMethodsCache.scan deliberately pre-filters on `beanFactory.getType(name, false)` before calling `getBean(name)` to avoid a circular construction: the cache itself is a Spring bean and `McpToolsService`/`McpPromptsService`/`McpResourcesService` all depend on it, so eagerly instantiating every Object-typed bean from inside the cache's factory method would deadlock. The BeanMethod record also captures `beanName` (in addition to `bean` and `method`) because the autoconfig startup log lines include the bean name, and we no longer have the `(name, Object)` entry-set that the old `getBeansWithAnnotation` path provided. Sort order is `(bean.getClass().getName(), method.getName())` ascending; previously each helper sorted methods by name within a single bean, and the old `getBeansWithAnnotation` iteration order was an implementation detail of Spring's bean-map ordering. The new order is deterministic, but downstream tests that asserted a specific numeric position in the tool/prompt/resource list (none observed in this repo) would need updating. The AOT processor's gating changed from "class has @ToolService/@PromptService/@ResourceService" to "class hosts at least one method-level annotation"; semantically equivalent for anyone who followed the old convention (class-level marker always paired with method-level annotations), but a user who somehow had a `@ToolService` class with zero `@McpTool` methods would no longer get hints registered — that's effectively a bug fix (nothing to register hints *for*). The three handler-kind autoconfigs flipped `before` → `after` on their `@AutoConfiguration` ordering hint so the `HandlerMethodsCache` bean (declared in `MocapiServerAutoConfiguration`) is available when they need to inject it.
