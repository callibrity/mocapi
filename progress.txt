WHAT_WAS_DONE: Added new module `mocapi-actuator-spring-boot-starter`: a Spring Boot Actuator endpoint at `/actuator/mcp` returning a read-only JSON snapshot of the MCP inventory on this node (server info block, counts, per-kind descriptor lists for tools/prompts/resources/resource-templates). Tool entries carry SHA-256 input/output schema digests (`sha256:<hex>`) rather than full schema bodies. `McpActuatorEndpoint` is `@Endpoint(id = "mcp")` with one `@ReadOperation`; `MocapiActuatorAutoConfiguration` is guarded by `@ConditionalOnClass(Endpoint.class)` + `@ConditionalOnAvailableEndpoint(endpoint = McpActuatorEndpoint.class)`, so users opt in via `management.endpoints.web.exposure.include=mcp`. Constructor takes `Implementation` (for server name/version — `BuildProperties` feeds the version via existing `mocapi-server` wiring, or the field is omitted when absent), plus the three paginated services (`McpToolsService`, `McpPromptsService`, `McpResourcesService`). Protocol version is a static constant (`2025-11-25`) representing the build's target MCP spec. Added `allResourceDescriptors()` / `allResourceTemplateDescriptors()` accessors to `McpResourcesService` (the spec assumed a separate `McpResourceTemplatesService` that doesn't exist — template descriptors live on the resources service). Response records (`McpActuatorSnapshot` + nested `ServerInfo` / `Counts` / `ToolInfo` / `PromptInfo` / `PromptArgumentInfo` / `ResourceInfo` / `ResourceTemplateInfo`) are public for downstream assertion. Schema digest helper is package-private static in `McpActuatorSnapshots`. Tests: `McpActuatorEndpointTest` (3 tests — happy path, null-schema → null digest, digest determinism), `MocapiActuatorAutoConfigurationTest` (2 tests — `ApplicationContextRunner` with + without `management.endpoints.web.exposure.include=mcp`), `McpActuatorEndpointExposedMockMvcTest` + `McpActuatorEndpointNotExposedMockMvcTest` (2 SpringBootTest MockMvc tests asserting 200 + JSON shape vs. 404, driven by a shared `ActuatorMockMvcTestApp` that excludes the mocapi-server autoconfigs — they need Substrate beans this test doesn't care about). NEXT_SPEC: 186-consolidate-feature-starters-into-autoconfigure.md
NOTES: The spec referenced a nonexistent `McpResourceTemplatesService` bean — in reality `McpResourcesService` owns both resources and resource templates. Adapted the endpoint constructor to take the single `McpResourcesService` and added two public accessors (`allResourceDescriptors()` / `allResourceTemplateDescriptors()`). The MockMvc test couldn't use plain `@SpringBootApplication` + `@EnableAutoConfiguration`: that triggers the mocapi-server autoconfigs which autowire an `AtomFactory` from Substrate. Chose to exclude those five server autoconfigs (`SubstrateOrderingAutoConfiguration`, `MocapiServerToolsAutoConfiguration`, `MocapiServerPromptsAutoConfiguration`, `MocapiServerResourcesAutoConfiguration`, `MocapiServerAutoConfiguration`) via `@EnableAutoConfiguration(exclude=...)` rather than enumerating every web/actuator autoconfig — Spring Boot 4 has restructured its autoconfigure packages (jackson moved to `spring-boot-jackson`, web servlet autoconfigs moved to `spring-boot-webmvc`, etc.) and an explicit include list is brittle. The endpoint publishes a static `PROTOCOL_VERSION = "2025-11-25"` constant rather than reading from `DefaultMcpServer.KNOWN_PROTOCOL_VERSIONS` (which is package-private and lists all supported versions, not "the current one") — if/when mocapi-server grows a public "current protocol version" constant, this endpoint can switch to it.
