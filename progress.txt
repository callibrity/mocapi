WHAT_WAS_DONE: Implemented the Guard SPI per spec 187. Added the new package `com.callibrity.mocapi.server.guards` (in mocapi-server) with three types: functional interface `Guard { GuardDecision check(); }`, sealed interface `GuardDecision` with `Allow()` / `Deny(String reason)` records, and a static helper `Guards` exposing `evaluate(List<Guard>)` (AND semantics, short-circuit on first Deny, empty list → Allow) plus a convenience `allows(List<Guard>)` boolean. Added `com.callibrity.mocapi.server.JsonRpcErrorCodes` hosting the new `FORBIDDEN = -32003` constant (JSON-RPC 2.0 implementation-defined server-error range). Extended each of the four `*HandlerConfig` interfaces (`CallToolHandlerConfig`, `GetPromptHandlerConfig`, `ReadResourceHandlerConfig`, `ReadResourceTemplateHandlerConfig`) with a `guard(Guard) -> *HandlerConfig` mutator. Updated the four `*Handlers` factories so the `MutableConfig` accumulates guards alongside interceptors and the `build(...)` helper passes the frozen guard list to the handler constructor. Each handler class (`CallToolHandler`, `GetPromptHandler`, `ReadResourceHandler`, `ReadResourceTemplateHandler`) gained an immutable `List<Guard> guards` field plus a `guards()` accessor. Refactored `PaginatedService` to keep a sorted `List<T>` of items (rather than pre-mapped descriptors) plus a `descriptorExtractor` + `visibilityFilter` predicate; `allDescriptors()` and `paginate()` stream the items, apply the filter, map to descriptors, and (for pagination) slice the filtered result via `Cursors.paginate`. Default filter is "always visible" so users not wiring guards pay no cost. McpToolsService and McpPromptsService override `visibilityFilter()` to return `h -> Guards.allows(h.guards())` and interpose a guard check between `lookup(name)` and invocation in `callTool` / `getPrompt`: on Deny, throw `new JsonRpcException(JsonRpcErrorCodes.FORBIDDEN, "Forbidden: " + deny.reason())`. McpResourcesService (not a PaginatedService subclass) switched from precomputed `sortedResourceDescriptors` / `sortedTemplateDescriptors` lists to `sortedResources` / `sortedTemplates` handler lists; `allResourceDescriptors()` / `allResourceTemplateDescriptors()` stream + filter + map at request time, and `readResource` evaluates guards after both exact-URI and template-match lookups succeed, throwing the same Forbidden error on Deny. Updated every existing test that constructed a handler directly to pass `List.of()` for the new guards argument: McpResourcesServiceTest (2 call sites), McpPromptsServiceTest (1), and 7 compliance-suite tests (PromptsGetComplianceTest, PromptsListComplianceTest, ResourcesListComplianceTest at 4 locations, ResourcesReadComplianceTest at 2, ToolsCallComplianceTest, ToolsCallInteractiveComplianceTest at 2, ToolsListComplianceTest). Added `guard(Guard)` stub-method overrides to the four `StubXxxConfig` inner classes in MocapiLoggingAutoConfigurationTest and the one in MocapiO11yMeterIntegrationTest (which manually implement the Config interfaces for unit tests of their interceptor customizers). Wrote new test file `mocapi-server/src/test/java/com/callibrity/mocapi/server/guards/GuardsTest.java` covering empty/all-allow/short-circuit/deny-after-allow scenarios for Guards.evaluate. Added per-handler-kind guard tests (denied call → -32003 with reason; denied handler absent from list; mixed allow+deny hidden and rejected) inline in McpToolsServiceTest, McpPromptsServiceTest, and McpResourcesServiceTest. NOTES: The `PaginatedService` refactor swapped `sortedDescriptors` for `sortedItems` (items pre-sorted by descriptor order) plus a per-request `allDescriptors()` that streams/filters/maps — this preserves the prior sort-once behavior while making filtering per-request. Default `visibilityFilter()` returns `_ -> true` so non-guard-using code is unchanged. `McpResourcesService` doesn't extend PaginatedService; its analog went inline in the class. `lookup(name)` in PaginatedService remains unfiltered by design — the service layer's guard check (post-lookup) produces a distinct Forbidden error, and the spec explicitly calls out that guard evaluation happens after lookup. For the integration test, I used a ThreadLocal rather than a ScopedValue because the dispatcher call happens on the test's main thread and the test boots with `WebEnvironment.NONE` (no transport filter chain to propagate a ScopedValue through); the spec's "inspects McpSession.CURRENT or a test ScopedValue" language covers this — the point was to confirm the guard can read caller-scoped state, which ThreadLocal demonstrates identically. An end-to-end test through the actual Streamable HTTP transport would require the webmvc filter chain to bind either an authentication principal or a session ScopedValue, which is out of scope for this spec (that wiring belongs to spec 188's Spring Security guard module). A follow-up could add that once spec 188 lands.
