NOTES: The `@CurrentTenant`-style tests annotate the target parameter with `@Nullable` so the default input-schema generator does not mark it required — otherwise `handler.call(emptyArgs)` fails input-schema validation before the resolver runs. This is a real limitation of the integration pattern (user resolvers do not hide their parameter from the generated schema); tolerable for now but worth a future spec if `@CurrentTenant`-type annotations become common. Each `resolver(...)` declaration uses the same type parameter as the existing `interceptor(...)` declaration (`? super JsonNode`, `? super Map<String, String>`, `? super Object`) so wildcard behavior is symmetric across the three attachment points. `McpToolParamsResolver` in the server-tools package is now public — if future work moves `@McpToolParams` handling behind a more abstract SPI, the promotion to public is the breaking bit to revisit. `ReadResourceHandlers.build` signature did not change (zero structural resolvers means the spec's "simplify by one parameter" language landed as "simplify by zero" here); consistent with the "catch-alls only exist where a catch-all makes sense" design.
